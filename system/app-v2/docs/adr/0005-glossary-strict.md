# ADR-0005: 用語統一の厳格遵守

**Status**: Accepted
**Date**: 2026-04-26
**Deciders**: 🌸美桜・🍁秋美・全姉妹

## Context

旧 tripot v1 の用語ブレ実態：
- 「案件」74箇所 vs 「商談」18箇所（同概念で混在）
- 「顧客」14 vs 「クライアント」4 vs 「お客様」1（3表記混在）
- 「メンバー」35 vs 「社員」3
- ボタン動詞：「保存」2 / 「追加」4 / 「登録」0 / 「更新」0（統一なし）

データ集計時、同じ概念を別名で呼ぶと **double-count or 漏れ事故** の温床。

## Decision

**`tripot-v2-glossary.md` で定義した用語を全層（DB schema / API / UI / コード）で厳格遵守する。**

| 概念 | 統一用語 | DB/コード | UI日本語 |
|---|---|---|---|
| 取引対象 | `deal` | `deals` テーブル | 案件 |
| 取引先 | `customer` | `customers` テーブル | 顧客 |
| 人 | `member` | `members` テーブル | メンバー |
| To Do | `task` | `tasks` テーブル | タスク |
| 行動 | `action` | `actions` テーブル | 行動 |
| 申請 | `approval` | `approvals` テーブル | 承認 |

### 禁止用語

- ❌ 商談・ディール（→ 案件）
- ❌ クライアント・お客様（→ 顧客）
- ❌ 社員・スタッフ（→ メンバー）
- ❌ 「保存」（→ 「登録する」「更新する」）

### ボタンラベル統一

- 新規 → **「登録する」**
- 編集後 → **「更新する」**
- 削除 → **「削除」**
- リスト追加 → **「追加」**（メンバー追加など、文脈限定）

## Consequences

### Positive

- データ集計の整合性100%
- 13社展開時、用語ブレが新会社に拡散しない
- 4姉妹+執事間のコミュニケーションも明確

### Negative

- 既存の旧tripot v1 のデータ移行時に用語マッピングが必要
- メンバー教育コスト（ただし日本語UI なので最小）

## Enforcement

- DB schema 命名で物理的に強制（`商談` テーブルが存在しない）
- ESLint custom rule で禁止用語警告（将来）
- コードレビュー時にチェックリスト化（セバスチャン監視枠）
